# 交通应急处置策略生成系统 项目展示报告

## 1. 项目概述

本项目是一个面向交通应急指挥的智能决策系统，核心目标是将“本地知识图谱 + 向量检索 + LLM 编排”有效融合，支持交通突发事件的快速解析、约束检索、方案生成与审查闭环。

系统主线由以下模块构成：
- `Neo4j` 知识图谱：结构化管理交通事件、动作、资源、因果和约束规则
- `ChromaDB` 向量库：存储法规、预案、案例文本，提供语义检索
- `Google AI Studio` LLM：负责结构化抽取、多模态理解、策略生成和评审
- `Streamlit` 前端：提供网页交互界面，支持文本与现场图片输入

项目入口在 `src/app.py`，通过 `PipelineOrchestrator` 调度整个处理流程。

## 2. 项目核心架构

### 2.1 编排主链路

当前主链路如下：

1. `DispatcherAgent`：将用户文本或图像转化为结构化实体
2. `EntityMatcherAgent`：将抽取结果规范化到图谱事件节点
3. `RetrievalLogicAgent`：拉取 Neo4j 约束与 Chroma 向量证据
4. `CommanderAgent`：生成单一处置策略草案
5. `EvaluatorAgent`：对方案进行审查与评分，形成闭环

整个流程被 `PipelineOrchestrator` 统一调度，最终输出结构化策略与审查结果。

### 2.2 数据契约与类型统一

项目通过 `src/contracts.py` 定义了完整数据契约，保障模块间传递数据结构稳定：
- `IncidentInput`：原始输入文本与可选图片字节
- `ExtractedEntities`：事故类型、定级、天气、风险要素、车辆、位置特征、伤亡估计等
- `Neo4jConstraint`：图谱约束规则
- `ChromaEvidence`：向量检索返回证据
- `RetrievalContext`：检索结果上下文
- `StrategyDraft`：指挥策略草案
- `ReviewResult`：审查打分与缺失项
- `PipelineResult`：编排最终输出

此设计体现了项目中“多智能体不走自由对话、通过契约传递数据”的严谨架构思想。

### 2.3 运行方式

在网页端展示时，用户进入 `src/app.py` 的 Streamlit 页面，输入事故描述，可选上传事故照片。

前端会展示：
- 接警解析结果
- 图谱约束与向量证据统计
- 初版方案与最终方案
- 审查结果与风险提示
- 调试侧边栏（调试模式可查看底层数据）

## 3. 关键模块实现

### 3.1 `src/agents.py`

该模块定义了多智能体的执行逻辑，包括：
- 文本/图像解析提示模板
- 结构化抽取与定级规则
- 事件规范化匹配
- 方案生成与修订
- 审查评分机制

其中，`DispatcherAgent` 结合文本与图像提示完成事故信息提取；`RetrievalLogicAgent` 负责组织查询词并调用双库检索；`EvaluatorAgent` 依据规则判断方案是否通过。

### 3.2 `src/retrieval_logic.py`

这是统一检索层，封装了双库检索逻辑：
- Neo4j 知识图谱连接：
  - 环境变量 `NEO4J_URI`、`NEO4J_USER`、`NEO4J_PASSWORD`、`NEO4J_DATABASE`
  - 查询事件节点、扩展因果关系、提取图谱约束
- ChromaDB 向量检索：
  - 默认目录 `./chroma_data`
  - 默认集合 `traffic_documents_v2`
  - 生成语义查询词并检索相关法规与预案文本

该模块还实现了：
- 事件别名匹配
- 语义查询构造
- 场景词提取与证据增强

### 3.3 `src/vectorDB.py`

负责向量数据库的构建与检索，关键点包括：
- 只处理 `data_raw/*.txt` 文本文件
- 默认分块长度 500 字符，避免向量检索粒度过大
- 使用 ChromaDB 本地持久化储存
- 支持添加文本块、查询检索、统计信息

这一部分是项目中“法规与案例知识库”的核心，实现了从文本到向量检索的完整闭环。

### 3.4 `src/llm_provider.py`

该模块统一封装 LLM 接口：
- 默认提供 `google_ai_studio`
- 默认模型 `gemma-3-27b-it`
- 支持环境变量覆盖 `TRAFFIC_LLM_PROVIDER`、`TRAFFIC_LLM_MODEL`
- 自动读取 `.env` 文件中的配置
- 兼容图像输入和 JSON 输出模式
- 带重试策略，支持 HTTP 错误重试和退避

这一部分体现了项目对远程大模型调用的工程化封装。

## 8. 可能的老师提问与参考回答

### 8.1 “你这个系统解决了什么实际问题？”

回答：本系统将道路交通突发事件从“文本/图片描述”自动转化为“可执行的应急处置策略”，中间引入国家交通应急预案图谱与法规文本检索，避免方案仅依赖模型自由生成，提高指挥决策的合规性与可执行性。

### 8.2 “为什么要用 Neo4j 和 ChromaDB 双库？”

回答：Neo4j 擅长表示结构化事件、因果关系和约束规则，适合提供硬性逻辑约束；ChromaDB 擅长语义检索法规与案例文本，适合提供软证据与上下文支持。双库协同可以实现“规则+证据”双重保障。

### 8.3 “系统如何保证生成方案的安全性和可执行性？”

回答：本项目设计了 `EvaluatorAgent` 审查环节，使用规则化评分输出 `executability_score`、`safety_score`、`compliance_score`，并对缺失关键动作、违反约束的情况进行反馈。此外，系统支持多轮修订，若首轮结果不合格会自动调整。

### 8.4 “你是如何处理图像输入的？”

回答：当用户上传现场照片时，`DispatcherAgent` 会调用视觉提示模板将图像内容转化为结构化标签，如危险要素、车辆类型和位置特征，然后合并到事故场景词中，进一步增强检索与方案生成能力。

### 8.5 “你做了哪些工程优化？”

回答：
- 统一了 LLM 提示与输出接口，避免每次调用都重写逻辑
- 使用环境变量配置模型、数据源、调试模式，支持可配置运行
- 设计了多条校验脚本，便于调试和回归验证
- 前端展示结构化结果，便于评审和交互

### 8.6 “这个项目下一步可以拓展哪些方向？”

回答：
- 增加更完整的图谱节点与事件类型覆盖
- 引入本地离线大模型，减少对远程 API 的依赖
- 加入多方案比较与最优方案选择
- 支持事故过程模拟与多部门协同任务分配

## 9. 总结

本项目已经实现了一个较为完整的交通突发事件智能决策系统，涵盖：接警解析、多模态输入、知识图谱检索、法规向量证据检索、单方案生成、规则化审查与网页展示。
