# E-KELL 交通应急处置系统 项目展示报告（面向 LLM 理解）

## 1. 报告目标与适用范围

本报告的目标不是“宣传项目”，而是为 LLM 提供一份可直接使用的项目事实说明，帮助其在后续开发、评测、排障、文档生成时做到：

- 准确理解当前系统真实实现，而非历史设想。
- 明确哪些能力已经落地，哪些仍是待优化项。
- 明确运行模式、数据链路、评测口径与常见风险点。

本报告基于当前代码与已有评测产物整理，重点对应目录：

- `src/`（主链路实现）
- `Tools/` 与 `Tools/importData/`（验证与导入流程）
- `experiments/results/`（评测产物）

---

## 2. 项目一句话定义

本项目是一个“图谱约束 + 向量证据 + 多 Agent 编排 + 审查修订”的交通事故应急处置生成系统，支持文本与图片输入，输出结构化处置策略与审查结论。

---

## 3. 当前系统现状（必须优先掌握）

### 3.1 主链路架构（已落地）

当前生产主链路由 `PipelineOrchestrator` 统一调度，核心流程为：

1. `DispatcherAgent`：图文抽取事故实体与定级信息。
2. `EntityMatcherAgent`：将事故表达规范到事件节点（含别名与重排）。
3. `RetrievalLogicAgent`：统一执行 Neo4j + Chroma 双检索。
4. `CommanderAgent`：生成单一处置方案，必要时按评审意见修订。
5. `EvaluatorAgent`：输出结构化评分与问题清单。

并非固定单模式执行：`orchestrator` 已支持路由分流与模式切换（见 3.2）。

### 3.2 执行模式（已实现，且影响行为）

通过环境变量 `TRAFFIC_AGENT_MODE` 控制：

- `auto`（默认）：先由 `RouterAgent` 判难度与置信度，再选链路。
- `single_agent`：单智能体一体化（抽取+检索+生成），默认跳过审查。
- `multi_no_review`：多 Agent 但跳过审查。
- `multi_with_review`：多 Agent + 审查 + 迭代修订（当前完整链路）。

`auto` 路由核心策略（已写入代码）：

- easy：优先走 `single_agent`。
- medium/hard 或低置信度：走 `multi_with_review`（即 G5 链路）。
- 无效路由目标或置信度不足时会触发回退保护。

### 3.3 审查修订机制（当前重点能力）

`multi_with_review` 下，系统不是“审一次就结束”，而是带门控的修订循环：

- 支持多轮修订（默认上限可配置，模式可分开配置）。
- 采用候选接收判据，避免“修订后反而退化”。
- 含质量守卫与大幅重写守卫，降低口号化回归。
- 含停机条件：
  - 达到分数阈值通过。
  - 达到直接通过高分阈值。
  - 达到修订轮数上限。
  - 连续停滞或连续无效修订触发停止。

结论：当前系统已经具备“生成-评审-受控修订-停机决策”的闭环逻辑，不是单次生成器。

### 3.4 数据契约（当前链路刚性约束）

Agent 间传递不走自由对话，而是以 `contracts.py` dataclass 作为统一契约：

- 输入：`IncidentInput`
- 抽取：`ExtractedEntities`、`MatchedNode`、`CasualtyEstimate`
- 检索：`Neo4jConstraint`、`ChromaEvidence`、`RetrievalContext`
- 生成与评审：`StrategyDraft`、`ReviewResult`
- 总输出：`PipelineResult`

这是系统可维护性与可评测性的关键基础。

## 4. 工作内容全量说明（做了什么）

### 4.1 多智能体编排落地

已完成从单点调用到标准编排器的工程化重构：

- 将抽取、匹配、检索、生成、审查拆分为独立 Agent。
- 在编排器内统一模式选择、重试、修订与收敛逻辑。
- 在输出层统一 `PipelineResult`，便于前端展示与离线评测。

### 4.2 事件规范化与匹配能力落地

`EntityMatcherAgent` 已实现：

- 别名库索引与候选短列表构建。
- 相似度融合（字符序列 + n-gram + 文字命中特征）。
- 层级偏置与重排，降低上位泛化节点误命中。

这使后续图谱检索不再依赖原始口语事故名。

### 4.3 双库检索整合落地

`DualRetrievalService` 统一封装：

- Neo4j：匹配事件、扩展事件、抽取约束（动作、实施主体、资源、预警/响应）。
- Chroma：构造证据查询串并返回文本证据。
- 两者合并进 `RetrievalContext`。

检索模式可切换：`dual | neo4j | chroma`。

### 4.4 生成与修订策略落地

`CommanderAgent` 已形成两段式能力：

- `generate`：按上下文产出初版策略。
- `revise`：按评审问题执行补丁式修订，尽量保留有效步骤。

且包含：

- 弱步骤识别与可执行化处理。
- 兜底方案生成（LLM 失败时保证可返回）。
- 按 mode/profile 注入不同 generation objective。

### 4.5 评审器落地

`EvaluatorAgent` 已支持三种审查模式：

- `llm_only`
- `rules_only`
- `hybrid`

输出包含：

- 三维评分与总分。
- 缺失动作、违规约束、风险提示。
- 可执行补丁建议（用于修订输入）。

### 4.6 前端展示链路落地

`src/app.py` 已对接编排器，支持：

- 文本输入 + 图片上传。
- 路由信息、抽取结果、检索统计、初版/终版策略、审查结果展示。
- 调试模式下展示底层结构化数据。

### 4.7 导入与治理流程落地

`Tools/importData/README.md` 已定义并支持标准导入 SOP：

1. 抽取（实体/关系）。
2. 质量门控与正式化。
3. Neo4j 导入（带 `source_tag`，支持回滚定位）。
4. Chroma 重建/更新。
5. 双库一致性核验。

并明确了严格规则：导入后缺失端点关系应为 0。

### 4.8 回归与评测工具链落地

已具备完整的检查脚本集，包括：

- Agent 级：`check_dispatcher_agent.py`、`check_retrieval_agent.py`、`check_commander_agent.py`、`check_evaluator_agent.py`。
- 编排级：`check_orchestrator.py`、`check_pipeline_regression.py`。
- 评测级：`experiments/results/` 下多批次结果文件。

结论：在该评测口径下，G5 平均分最高。

### 5.2 按难度分组（G3 vs G5）

来自 `experiments/results/G3_G5_各难度得分统计结果/avg_score.csv`：

- easy：G3=`0.850192`，G5=`0.850000`
- medium：G3=`0.847432`，G5=`0.858784`
- hard：G3=`0.833443`，G5=`0.844598`

结论：中高难场景 G5 相对优势更明显，符合“审查修订链路对复杂场景收益更高”的预期。

### 5.3 当前两组结果快照（仓库现有 CSV 直接统计）

基于 `G5_full.csv` 与 `single_none_full.csv` 的现有数据统计：

- `G5_full.csv`
  - 样本数：109
  - 平均总分：`0.6768`
  - 平均规则分：`0.5678`
  - 平均时延：`103668.72 ms`
- `single_none_full.csv`
  - 样本数：109
  - 平均总分：`0.8942`
  - 平均规则分：`0.7670`
  - 平均时延：`15271.17 ms`

解释要点：

- 该快照受运行批次状态影响较大（模型可用性、评审网络稳定性、样本分布）。
- 因此必须结合“同口径对照实验”解读，不可孤立比较不同批次。

### 5.4 已暴露并被记录的问题类型

从结果文件可见的主要失败来源包括：

- 远程 LLM 网络异常（如 SSL EOF）导致审查质量下降或回退。
- 轻微事故场景下的过度响应倾向（资源错配）。
- 个别样本的修订停机触发（连续无效修订）。

这类问题已在代码中有防护策略，但仍是后续优化重点。

## 6. 运行与依赖事实

### 6.1 推荐环境

- OS：Windows
- Python：3.10+
- Conda 环境：`traffic_env`

### 6.2 外部依赖

- Neo4j：默认 `bolt://localhost:7687`
- ChromaDB：本地目录 `./chroma_data`
- LLM Provider：
  - 默认 `google_ai_studio`
  - 默认模型 `gemma-3-27b-it`
  - 兼容 `xfyun_maas` provider（用于部分评测口径）

### 6.3 常用环境变量

- LLM：`TRAFFIC_LLM_PROVIDER`、`TRAFFIC_LLM_MODEL`、`GEMINI_API_KEY`/`GOOGLE_API_KEY`
- 路由/模式：`TRAFFIC_AGENT_MODE`
- 检索：`TRAFFIC_RETRIEVAL_MODE`
- Neo4j：`NEO4J_URI`、`NEO4J_USER`、`NEO4J_PASSWORD`、`NEO4J_DATABASE`

### 6.4 启动方式

```powershell
conda activate traffic_env
pip install -r requirements.txt
streamlit run src/app.py
```

## 7. 系统边界与已知限制（LLM 必须知道）

- 当前输出是“单方案生成+审查修订”，不是多方案博弈与自动择优。
- 方案质量显著依赖：
  - 路由判定准确性；
  - 检索证据命中质量；
  - 远程评审稳定性。
- 轻微事故场景仍可能出现过度处置倾向，需要继续强化“分级匹配与资源克制”。
- 评测结论必须注明批次与口径，避免跨批次误对比。

---

## 8. 给后续 LLM 的协作指引

若你是后续接手本项目的 LLM，请遵循：

1. 先按 `src/orchestrator.py -> src/agents.py -> src/retrieval_logic.py` 理解主链路，不要只看旧文档。
2. 修改策略前，先确认当前 `TRAFFIC_AGENT_MODE` 与评测口径。
3. 涉及数据导入，严格按 `Tools/importData/README.md` 的 SOP 执行。
4. 给出优化建议时，必须区分：
   - 架构建议（长期）
   - 参数调优（短期）
   - 已在代码中存在的机制（避免重复造轮子）

## 9. 总结

项目已完成从“单次文本生成”到“可路由、可检索、可审查、可修订、可评测”的交通应急决策系统工程化落地。

当前最核心价值不在于某一次样本分数，而在于：

- 架构闭环已建立；
- 数据契约已稳定；
- 工具链与评测链齐备；
- 关键风险点已可定位、可迭代优化。

这使项目具备持续演进到“高可靠交通应急决策系统”的基础条件。
