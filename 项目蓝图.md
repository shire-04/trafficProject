# 交通应急 DSS 项目蓝图（当前版本）

## 目标

当前项目的目标是构建一个面向交通事故与突发事件的单方案应急决策支持系统，主链路为：

`文本/图片输入 → 结构化抽取 → Neo4j + Chroma 双库检索 → 单方案生成 → 审查闭环 → 输出最终策略`

本阶段只实现**单方案**，不做多方案对比。

## 当前正式架构

### 1. 输入层
- 用户输入事故文本描述。
- 可上传现场图片。
- 图片当前已接入输入契约，但新链路中的视觉分析仍处于轻量占位阶段。

### 2. 调度与编排层
- `src/orchestrator.py`
- 核心类：`PipelineOrchestrator`
- 当前流程：
  1. `DispatcherAgent` 提取结构化实体
  2. `RetrievalLogicAgent` 统一调用 Neo4j 与 ChromaDB
  3. `CommanderAgent` 生成单方案草案
  4. `EvaluatorAgent` 审查方案
  5. 如首次未通过，自动执行一次修订后重审

### 3. Agent 分工
- `DispatcherAgent`：抽取事故类型、天气、危险要素、车辆、位置、伤亡估计。
- `RetrievalLogicAgent`：负责双库检索与上下文整合。
- `CommanderAgent`：根据约束和证据生成单方案，并在需要时执行一次修订。
- `EvaluatorAgent`：基于规则检查缺失动作、风险点与资源覆盖情况。

### 4. 数据层

#### Neo4j
- 独立实例：`traffic_plan_v2`
- 当前使用数据库：`neo4j`
- 当前正式图谱来源：`data_clean/neo4j_import/*.csv`
- 实体身份规则：**以 `id` 为准，不以 `name` 为准**

#### ChromaDB
- 正式 collection：`traffic_documents_v2`
- 当前正式文本范围：`data_raw/国家交通应急预案.txt`
- 当前用途：为方案生成与审查提供法规/预案证据，不替代图谱硬约束

## 当前原则

- 小步推进
- 每一步都可验证
- 不随意扩展正式双库范围
- 不恢复旧会话式角色链路
- 新增能力优先落在 `contracts.py`、`retrieval_logic.py`、`agents.py`、`orchestrator.py` 这条主线

## 一句话总结

当前项目已经切换到以 `PipelineOrchestrator` 为核心的结构化单方案闭环架构，后续演进应全部基于这条新主线继续推进。
