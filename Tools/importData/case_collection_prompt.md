# 经典交通事故案例采集提示词

这份提示词用于交给外部 LLM，让它帮你收集**信息全面、准确、便于后续实体/关系抽取**的交通应急处置案例。

它已经对齐当前项目的文本解析格式：

- 每个案例之间必须空一行
- 每个案例必须严格由 4 行组成
- 第 1 行是标题
- 第 2 行必须以 `特定事故：` 开头
- 第 3 行必须以 `后果：` 开头
- 第 4 行必须以 `措施：` 开头

这样可以直接用于 `Tools/importData/extract_case_entities_to_csv.py` 和 `Tools/importData/extract_case_relations_to_csv.py`。

---

## 推荐主提示词

```text
你是一名“交通事故案例资料整理员 + 应急处置知识抽取前置整理员”。

你的任务是：
为我整理若干个“经典交通应急处置案例”，要求内容真实风格、信息充分、结构稳定，并且便于后续做实体抽取与关系抽取。

请严格遵守以下要求：

一、总体目标
1. 输出的案例要尽量具备较高信息密度，能够支持后续抽取以下要素：
   - 事故事件（Event）
   - 处置动作（Action）
   - 处置部门（Department）
   - 资源/装备/力量（Resource）

2. 输出的案例必须尽量体现“事故 -> 后果 -> 处置措施”的清晰链条，便于后续抽取关系，例如：
   - CAUSES
   - TRIGGERS
   - REQUIRES
   - IMPLEMENTED_BY

3. 案例应覆盖典型交通应急场景，例如：
   - 普通道路交通事故
   - 高速公路多车追尾
   - 隧道事故
   - 桥梁事故
   - 危化品运输事故
   - 恶劣天气诱发事故
   - 起火、泄漏、爆燃、坠桥、塌陷、滑坡等复合型事故

二、真实性与准确性要求
1. 尽量采用“接近真实事故通报/案例摘要”的写法，不要写成小说。
2. 不要编造明显违背常识的装备、部门、法规、处置措施。
3. 如果不能确认具体数字，可以用“多人受伤”“部分人员被困”“交通中断数小时”等稳健表达。
4. 法律依据、预案依据必须写成规范名称，不要写成模糊的“根据相关规定”。
5. 措施必须尽量细化到可执行动作，不要只写“有关部门妥善处置”。

三、信息完整性要求
每个案例都应尽可能包含以下内容中的大部分：
1. 事故发生场景：道路类型、地点特征、天气、时间特征、车辆类型
2. 核心事故机制：碰撞、侧翻、泄漏、起火、追尾、坠落、冲撞、塌陷、阻断等
3. 后果：伤亡、被困、道路中断、车辆滞留、环境污染、次生灾害、社会影响
4. 处置措施：
   - 现场警戒/封控/交通管制
   - 人员搜救与医疗救治
   - 消防灭火/防化处置/围堵堵漏/排烟通风
   - 分流绕行/恢复通行/抢修保通
   - 专业队伍、装备、吊车、拖车、消防、环保、医疗、交警、应急、交通部门等资源调用
   - 信息发布、风险监测、后续调查等

四、输出格式要求（必须严格遵守）
你输出的每一个案例都必须严格只有以下 4 行，不能多行，不能少行，不能改字段名：

第1行：案例标题
第2行：特定事故：<这里写事故经过与场景>
第3行：后果：<这里写事故造成的后果>
第4行：措施：<这里写应急处置措施>

额外要求：
1. 每个案例之间必须空一行。
2. 不要加编号，如“1.”、“案例1”。
3. 不要输出 Markdown 表格。
4. 不要输出项目符号。
5. 不要输出 JSON。
6. 不要输出任何解释、备注、总结、提示语。
7. 不要在正文外再写“以下是案例”“好的”等客套话。

五、字段内容写法要求
1. 标题要简洁明确，像真实事故标题，例如：
   - 京港澳高速湘潭段多车连环相撞事故
   - 隧道内危化品车追尾泄漏事故

2. “特定事故：”这一行必须重点描述：
   - 在什么地方发生
   - 涉及什么车辆或危险源
   - 事故是怎么发生的
   - 必要时提到天气、路况、能见度、地形、道路设施

3. “后果：”这一行必须重点描述：
   - 人员伤亡或被困情况
   - 交通中断、拥堵、封闭、滞留情况
   - 环境污染、爆炸风险、火灾蔓延、桥梁受损、隧道高温、道路塌陷等次生风险

4. “措施：”这一行必须重点描述：
   - 哪些部门/队伍参与
   - 采取了哪些具体动作
   - 是否引用法律法规或应急预案
   - 是否调用专业装备、专业力量、监测手段、转运、清障、绕行、排险、恢复通行等

六、质量控制要求
1. 不要写空泛套话，如“立即开展救援并做好善后工作”。
2. 尽量把多个措施写成一个有逻辑顺序的长句，可用“；”“+”“并”“同步”“随后”等连接。
3. 避免同一批案例只换地名不换内容，要保证案例之间有场景差异。
4. 案例的重点必须是“交通应急处置”，不是单纯事故新闻。
5. 优先选择便于抽取的表达，避免过多代词（如“其”“该情况”“有关方面”）。

七、生成数量要求
本次请输出【10】个案例。

请现在直接开始输出案例正文，严格遵守格式，不要添加任何额外说明。
```

## 更稳的补充约束版

如果你发现某个 LLM 经常输出跑偏，可以改用下面这版：

```text
请生成 10 个交通应急处置案例。

严格要求：
1. 每个案例必须只有 4 行。
2. 第 1 行只能是标题，不能带“标题：”字样。
3. 第 2 行必须以“特定事故：”开头。
4. 第 3 行必须以“后果：”开头。
5. 第 4 行必须以“措施：”开头。
6. 每个案例之间空一行。
7. 不允许输出任何说明文字、编号、表格、项目符号、JSON、Markdown 标题。
8. 每个案例都必须至少包含以下信息中的大部分：道路场景、车辆类型、事故机制、伤亡/被困、交通影响、处置措施、部门力量、法规或预案依据。
9. 措施不能空泛，必须写出具体动作。
10. 尽量使用真实事故通报风格的中文。

开始直接输出正文。
```

---

## 使用建议

- 先让 LLM 生成 `5-10` 个案例做小样检查，再批量生成。
- 优先检查：
  - 是否严格是 4 行结构
  - `措施：` 是否足够具体
  - 是否含有法规/预案依据
  - 是否容易拆出 `事故 -> 后果 -> 处置` 三段逻辑
- 如果某个模型总爱加解释，可以在提示词末尾再加一句：

```text
如果你输出了案例正文之外的任何内容，则视为失败，请重新仅输出案例正文。
```
