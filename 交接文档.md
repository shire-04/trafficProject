# 交接文档（G1~G5 实验链路）

## 1. 目的与口径

本文件用于将当前项目中 G1~G5 评测工作交接给下一位 LLM/开发者，重点覆盖：
- 各组定义
- 各组实际运行流程对应到的代码文件
- 配置读取入口
- 结果输出目录与命名方式
- 结果汇总/对比脚本

实事求是说明：
- G1~G5 的权威映射来自 `experiments/GROUP_DEFINITIONS.md`。
- 运行时并不存在 `g1.py` 这类单独实现文件；五组共用同一套编排与评测代码，通过 `retrieval_mode` 和 `agent_mode` 区分。

---

## 2. G1~G5 权威定义

来源：`experiments/GROUP_DEFINITIONS.md`

1. G1 = single + neo4j
2. G2 = single + chroma
3. G3 = single + dual
4. G4 = multi_no_review + dual
5. G5 = multi_with_review + dual

参数字段：
- retrieval_mode: neo4j | chroma | dual
- agent_mode: single | multi_no_review | multi_with_review | auto

## 3. 统一执行入口（五组共用）

### 3.1 实验主执行脚本
- `Tools/run_eval_experiments.py`

职责：
- 读取数据集（默认 `data_raw/评测数据集.jsonl`）
- 注入环境变量：
  - `TRAFFIC_RETRIEVAL_MODE`
  - `TRAFFIC_AGENT_MODE`
- 调用 `PipelineOrchestrator.run_once(...)` 逐条推理
- 调用 `Tools/eval_metrics.py` 计算规则分/LLM裁判分
- 输出逐条 CSV（含 total_score、llm_judge_*、notes 等）

### 3.2 核心编排器
- `src/orchestrator.py` -> `PipelineOrchestrator`

职责：
- 串联 Dispatcher -> Matcher -> Retrieval -> Commander -> Evaluator
- 根据 `TRAFFIC_AGENT_MODE` 走不同分支（默认 `auto`）
- 在 `multi_with_review` 下执行修订循环、停滞/无效修订检测
- 在 `auto` 下先由 RouterAgent 判定难度：easy -> G3 链路，medium/hard -> G5 链路

### 3.3 Agent 与 LLM调用
- `src/agents.py`
  - Dispatcher/Matcher/Commander/Evaluator 的实现
  - 读取各阶段模型与超时配置（例如 `DISPATCHER_TEXT_MODEL`、`COMMANDER_TEXT_MODEL`、`EVALUATOR_TEXT_MODEL`）
- `src/llm_provider.py`
  - provider/model 统一入口
  - 从 `.env` 加载 `env:` 前缀变量
  - 支持 `google_ai_studio` 与 `xfyun_maas`

### 3.4 检索层
- `src/retrieval_logic.py`
  - 读取 `TRAFFIC_RETRIEVAL_MODE`
  - Neo4j 连接参数：`NEO4J_URI/USER/PASSWORD/DATABASE`
  - Chroma 集合默认来自 `src/vectorDB.py` 的生产集合常量

### 3.5 评分层
- `Tools/eval_metrics.py`

职责：
- 基于 must_actions / must_constraints / must_evidence_topics 计算规则评分
- 按 `EVAL_SCORE_BACKEND` 决定 rules/hybrid/llm
- 可调用外部评委 LLM（`EVAL_ENABLE_LLM_JUDGE=1`）

## 4. 各组流程与涉及文件

## 4.1 G1（single + neo4j）

运行参数：
- `--group-id G1 --retrieval-mode neo4j --agent-mode single`

实际流程（代码层面）：
1. `Tools/run_eval_experiments.py` 设置 `TRAFFIC_RETRIEVAL_MODE=neo4j`, `TRAFFIC_AGENT_MODE=single`
2. `src/orchestrator.py` 中 `single` 会被归一到 `single_v2`
3. `single_v2` 流程：
   - Dispatcher 抽取（`src/agents.py`）
   - Matcher 匹配（`src/agents.py`）
   - Retrieval（仅图谱分支，`src/retrieval_logic.py`）
   - Commander 生成（`src/agents.py`）
   - 不执行评审与修订（编排器给出实验模式说明）
4. 结果送入 `Tools/eval_metrics.py` 打分后落盘

该组关键文件：
- `Tools/run_eval_experiments.py`
- `src/orchestrator.py`
- `src/agents.py`
- `src/retrieval_logic.py`
- `Tools/eval_metrics.py`

## 4.2 G2（single + chroma）

运行参数：
- `--group-id G2 --retrieval-mode chroma --agent-mode single`

与 G1 唯一区别：
- `src/retrieval_logic.py` 走向量检索分支（Chroma）

其余流程、文件与 G1 相同。

## 4.3 G3（single + dual）

运行参数：
- `--group-id G3 --retrieval-mode dual --agent-mode single`

与 G1/G2 的区别：
- `src/retrieval_logic.py` 同时使用 Neo4j + Chroma 的 dual 分支

其余流程、文件与 G1 相同。

## 4.4 G4（multi_no_review + dual）

运行参数：
- `--group-id G4 --retrieval-mode dual --agent-mode multi_no_review`

流程：
1. Dispatcher -> Matcher -> Retrieval(dual) -> Commander
2. 在 `src/orchestrator.py` 的 `multi_no_review` 分支直接跳过 Evaluator 与修订
3. 输出草案并进入 `Tools/eval_metrics.py`

关键文件：
- `Tools/run_eval_experiments.py`
- `src/orchestrator.py`（`multi_no_review` 分支）
- `src/agents.py`
- `src/retrieval_logic.py`
- `Tools/eval_metrics.py`

## 4.5 G5（multi_with_review + dual）

运行参数：
- `--group-id G5 --retrieval-mode dual --agent-mode multi_with_review`

流程：
1. Dispatcher -> Matcher -> Retrieval(dual) -> Commander 初稿
2. Evaluator 审查（LLM/rules/hybrid 由配置决定）
3. 若未达条件进入修订循环：
   - Commander.revise
   - Evaluator.review
   - 基于 score_delta、停滞次数、最大轮次等判停
4. 输出最终方案并进入 `Tools/eval_metrics.py`

关键文件（相对 G4 新增的重点）：
- `src/orchestrator.py`（修订循环与停机条件）
- `src/agents.py`（Evaluator 审查、Commander 修订）

## 4.6 Auto（动态路由）

运行参数：
- `--agent-mode auto`

流程：
1. RouterAgent 根据输入文本/图片判定 `easy|medium|hard`
2. easy -> `single_agent`（G3 链路）
3. medium/hard -> `multi_with_review`（G5 链路）
4. 若路由判定失败或置信度不足，自动回退到 G5 链路

## 10. 修订链路优化设计计划（2026-04-06）

目标：在保持 G5 当前高基线的前提下，让修订链路呈现“净增益、低回退、可解释”的稳定行为。

### 10.1 设计原则（经典框架）

1. MAPE-K：先监测缺陷，再分析类型，再计划补丁，最后执行与沉淀知识。
2. CEGIS：由审查反例驱动修订，禁止无目标重写。
3. 风控门控：修订稿不满足接收条件时拒收，保持当前基线方案继续迭代。
4. 最佳保留：全过程保留 best_draft/best_review，最终输出历史最优而不是最后一稿。

### 10.2 执行项与落点

1. 新增“候选修订接收门控”：
- 位置：`src/orchestrator.py`
- 策略：对比 current_review 与 revised_review 的分数、子维度、缺失动作数、违规约束数。
- 规则：若候选稿未达到最小收益或出现明显退化，则拒收候选稿，不更新当前基线。

2. 新增可配置阈值（优先针对 multi_with_review）：
- `TRAFFIC_REVISION_ACCEPT_MIN_DELTA(_MULTI_WITH_REVIEW)`
- `TRAFFIC_REVISION_ACCEPT_SCORE_DROP_TOLERANCE(_MULTI_WITH_REVIEW)`
- `TRAFFIC_REVISION_ACCEPT_DIM_DROP_TOLERANCE(_MULTI_WITH_REVIEW)`

3. 调整修订循环状态转移：
- 拒收候选稿时，`current_draft/current_review/current_score` 保持不变。
- 拒收计入无效修订统计，触发无效停机阈值。

4. 验证方案：
- 固定当前最优口径：G5 + dual + 新默认目标。
- 对照运行：`rev=0` 与 `rev=5`（其余参数保持一致）
- 指标：平均分 delta、improve/worse 样本计数、回退样本原因。

### 10.3 执行状态

1. 计划写入：已完成。
2. 代码实现：已完成。
3. 对照评测：已完成。

### 10.4 本轮执行记录

代码改造：
1. 在 `src/orchestrator.py` 新增修订接收门控阈值读取：
- `TRAFFIC_REVISION_ACCEPT_MIN_DELTA(_MULTI_WITH_REVIEW)`
- `TRAFFIC_REVISION_ACCEPT_SCORE_DROP_TOLERANCE(_MULTI_WITH_REVIEW)`
- `TRAFFIC_REVISION_ACCEPT_DIM_DROP_TOLERANCE(_MULTI_WITH_REVIEW)`
2. 新增 `_is_revision_candidate_acceptable(...)` 候选接收判定。
3. 修订循环接入“候选拒收后保持当前基线”机制，并记录 `pipeline_revision_candidate_rejected/accepted` 调试事件。

对照实验：
1. 目录：`experiments/results/g5_revision_gate_eval_10_20260406_183951/`
2. rev0：`G5_10_rev0.csv`
3. rev5：`G5_10_rev5.csv`
4. 结果：
- 平均分：rev0=0.8160，rev5=0.8600，delta=+0.0440
- 样本计数：improve=6，worse=3，same=1

结论：
1. 在本轮门控策略下，修订链路相对 rev0 呈现显著净正收益。
2. 仍有少量回退样本，后续可围绕回退样本做分型约束（安全类/时序类/证据锚定类）进一步收敛。

### 10.5 第二轮加固执行记录（2026-04-06 19:25）

本轮目标：将“回退样本分型”转成可执行门控，优先压制高基线样本回退。

代码改造：
1. 在 `src/orchestrator.py` 增加高分保护阈值读取：
- `TRAFFIC_REVISION_HIGH_SCORE_PROTECT_FLOOR(_MULTI_WITH_REVIEW)`
- `TRAFFIC_REVISION_HIGH_SCORE_DROP_TOLERANCE(_MULTI_WITH_REVIEW)`
- `TRAFFIC_REVISION_HIGH_SCORE_SAFETY_DROP_TOLERANCE(_MULTI_WITH_REVIEW)`
2. 在 `_is_revision_candidate_acceptable(...)` 中新增“高分基线防回退”判定：
- 当当前分数高于保护阈值时，候选稿若总分回退则拒收（`high_score_regression`）
- 当当前分数高于保护阈值时，候选稿若安全分明显回退则拒收（`high_score_safety_worsened`）

对照实验（新代码下重跑）：
1. 目录：`experiments/results/g5_revision_gate_hardsafe_10_20260406_192508/`
2. rev0：`G5_10_rev0.csv`
3. rev5：`G5_10_rev5.csv`
4. 结果：
- 平均分：rev0=0.8420，rev5=0.8770，delta=+0.0350
- 样本计数：improve=6，worse=3，same=1

回退样本分型（本轮）：
1. S005（-0.08）：安全细节缺失型（对周边群众潜在风险评估与控制不足）
2. S007（-0.03）：安全细节缺失型（二次事故预防措施不够具体）
3. S010（-0.01）：场景关键风险锚定不足型（加油站爆炸风险应对不够直接）

阶段结论：
1. 第二轮加固后，修订链路仍保持显著净正收益。
2. 回退样本已从“高基线普遍回退”收敛到“安全细节/关键风险锚定不足”的窄类型。
3. 下一轮应把“关键风险锚定覆盖”做成结构化硬约束（优先加油站/爆燃/围观二次事故等高风险词槽与动作对齐）。

### 10.6 去场景化修正（2026-04-06 20:05）

修正动机：
1. 场景定制型硬约束存在过拟合风险，不符合“能力提升优先、泛化优先”的优化目标。
2. 本轮将该类约束撤回，改为仅依赖通用能力信号的门控。

代码修正：
1. 在 `src/orchestrator.py` 移除上一轮新增的高分保护硬门控参数与判定。
2. 保留通用非退化门控（总分容忍、子维度容忍、缺失动作/违规约束不恶化）。
3. 新增通用能力型联合退化拒收：当“可执行性 + 安全性”同时下降且合规不提升时拒收（`core_capability_joint_worsened`）。

对照实验（去场景化规则）：
1. 目录：`experiments/results/g5_revision_gate_general_10_20260406_200527/`
2. rev0：`G5_10_rev0.csv`
3. rev5：`G5_10_rev5.csv`
4. 结果：
- 平均分：rev0=0.8340，rev5=0.8520，delta=+0.0180
- 样本计数：improve=5，worse=4，same=1

结论：
1. 去场景化后仍保持净正收益，说明收益不依赖场景硬编码。
2. 增益幅度较“高分保护门控”有所回落，符合“更泛化、更保守”的预期。
3. 下一步优化重点应放在评审信号稳定性与多候选重排，而不是增加场景型规则。
