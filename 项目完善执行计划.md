## 4. 执行原则

后续所有工作继续遵守以下边界：

1. 不重新把业务规则硬编码回主链路
2. 不让图谱承担当前事故实例的自动定级职责
3. 不让 `DispatcherAgent` 重新承担标准化匹配职责
4. 可以增强别名、召回、日志、重试、可观测性，但不要把这些增强做成隐性规则系统
5. 先用固定案例验证，再改 prompt 或代码，不要凭直觉微调

---

## 5. 分阶段执行计划

### 阶段 A：重建 Google + Gemma 多案例回归基线（最高优先级）

#### 目标

确认当前 `.env + gemma3` 配置下，5 个固定案例的真实通过率与失败分布。

#### 关键任务

1. 运行 `Tools/check_pipeline_regression.py`
2. 保存输出结果，记录以下指标：
   - `success_count`
   - `non_empty_incident_type_raw_count`
   - `matched_event_case_count`
   - `non_empty_strategy_case_count`
   - `approved_case_count`
   - `handoff_case_count`
3. 对每个失败案例补充人工分析，判断失败发生在哪一环：
   - 抽取
   - 匹配
   - 检索
   - 生成
   - 审查

#### 交付物

- 一份新的多案例回归结果记录
- 一份失败案例清单
- 当前 `gemma3` 路线的可用性结论

#### 验收标准

- 5 个案例全部可执行完成，不出现脚本级崩溃
- 至少 4 个案例 `incident_type_raw` 非空
- 至少 3 个案例存在有效 `matched_events`
- 至少 3 个案例有非空方案步骤

---

### 阶段 B：专项优化 `EntityMatcherAgent` 排序质量

#### 目标

把“能匹配”提升为“Top1 更贴切”。

#### 关键任务

1. 统计回归案例中 Top1 与人工预期不一致的样例
2. 检查这些案例对应的 alias 是否不足或不够贴近真实口语表达
3. 检查当前 shortlist 是否过长，导致候选噪声过大
4. 评估是否需要：
   - 优化别名表内容
   - 优化候选裁剪逻辑
   - 优化 matcher prompt 中对 `aliases` 的权重描述

#### 交付物

- 一份 matcher 误匹配分析表
- 一轮别名表更新建议
- 一版更稳妥的 matcher 排序策略

#### 验收标准

- 回归案例中 Top1 的主观合理性明显提升
- 更具体的事故场景不再频繁被泛化节点抢走 Top1

#### 风险提示

- 不要把 alias 扩充演变成业务规则表
- 不要为了某几个案例“手工硬配”节点

---

### 阶段 C：优化检索 query 质量

#### 目标

在匹配结果更稳定后，提高图谱链条和 Chroma 证据的相关性。

#### 关键任务

1. 分析 `retrieval_result` 日志中的：
   - `query_term_count`
   - `expanded_event_count`
   - `constraint_count`
   - `evidence_count`
   - `evidence_query_preview`
2. 找出高质量案例与低质量案例的 query 差异
3. 优化 `build_query_terms()` 中以下信息的组合方式：
   - `incident_type_raw`
   - `matched_events`
   - `aliases`
   - `hazards`
   - `vehicles`
   - `location_features`
   - `severity`
4. 减少噪声词，避免 evidence query 过度冗长

#### 交付物

- 检索质量差异分析
- 一版更干净的 query 构造策略

#### 验收标准

- 低质量案例的 `evidence_count` 或相关性有所改善
- 生成阶段引用图谱动作和法规证据更加稳定

---

### 阶段 D：微调生成与审查边界

#### 目标

在匹配和检索相对稳定后，再提升方案质量和审查一致性。

#### 关键任务

1. 汇总常见问题：
   - 步骤过于空泛
   - 资源遗漏
   - 审查标准忽严忽松
   - `APPROVED` 但理由中仍包含明显缺失
2. 针对问题分别调优：
   - `STRATEGY_GENERATION_PROMPT`
   - `STRATEGY_REVIEW_PROMPT`
3. 保持“无业务规则兜底”的前提下提高输出一致性

#### 交付物

- 生成质量问题清单
- 审查一致性问题清单
- 一轮 prompt 微调结果

#### 验收标准

- 方案步骤更具执行顺序
- 审查结论与缺失动作更一致
- 人工阅读时的可用性明显提升

---

### 阶段 E：补充工程韧性与文档收尾

#### 目标

把当前“可运行”提升到“更适合持续迭代”。

#### 关键任务

1. 评估是否加入 Google 429/网络异常重试
2. 评估是否增加调用失败的统一错误类型
3. 更新 README 或单独模型配置文档，写清：
   - `.env` 支持的字段
   - 推荐模型
   - 已知限制
4. 继续弱化或整理旧模块的误导性表达：
   - `src/reasoning_engine.py`
   - `src/vectorDB.py`

#### 交付物

- 更清晰的运行说明
- 更稳妥的失败处理策略
- 旧模块风险说明

#### 验收标准

- 新接手者不用重新猜环境变量怎么配
- API 配额或网络波动时，系统不至于直接失控

## 6. 推荐执行顺序

为了避免分散精力，推荐严格按下面顺序推进：

1. **先跑 `Tools/check_pipeline_regression.py`**
2. **再分析 matcher Top1 排序问题**
3. **然后优化 retrieval query**
4. **最后再改 commander / evaluator prompt**
5. **回归通过后再做工程韧性和文档收尾**

不要一上来同时改 matcher、retrieval、prompt，否则无法知道哪一步真正起作用。
